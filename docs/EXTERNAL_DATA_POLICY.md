# External Data Policy

This policy governs how FlexSavvy interacts with external data sources, manages fixtures, and separates live API calls from automated testing.

## Offline Tests

All automated tests must run without internet access. This is non-negotiable.

- Unit tests import source modules directly and use in-memory data structures or committed fixtures.
- Integration tests use committed fixture files located under `fixtures/`.
- E2E tests serve only the local build output; no external network calls are permitted during test runs.
- If a test requires network access to pass, it is not an automated test — it is a manual validation step labelled separately in the task file.

## Fixture Provenance

Every committed fixture must have a clear and legitimate provenance:

| Fixture Type | Allowed Source | Requirements |
|-------------|----------------|--------------|
| Synthetic data | Generated by scripts in `scripts/` | Deterministic generation; seed values recorded |
| Public datasets | Openly licensed or CC0 data | License attribution in fixture directory README |
| Anonymised samples | Derived from real data with all identifiers removed | No meter IDs, no filenames, no addresses, no personal names |

- Each fixture directory contains a `README.md` describing the contents, source, and any transformation applied.
- Fixtures must be small enough to commit (individual files under 1 MB, total fixtures under 10 MB).
- Fixture data representing UK half-hourly intervals must include at least one DST boundary case.

## Allowed Public APIs

FlexSavvy may interact with the following public APIs in production, behind typed adapter interfaces:

| API | Purpose | Authentication Required? |
|-----|---------|--------------------------|
| Octopus Energy public tariff API | Tariff catalogue and rate retrieval | No (public endpoint) |
| GB Electricity System Operator carbon intensity API | Carbon data for scoring | No (public endpoint) |

- All API access is isolated behind adapter modules in `src/lib/adapters/`.
- Adapters define typed interfaces; implementations may swap live calls for fixture playback.
- No API credentials are stored in the repository, in environment variables checked into source control, or hardcoded.

## Live-Validation Separation

Live API validation is a separately labelled step within a task file:

1. **Automated tests** — run against committed fixtures only. Never call live APIs.
2. **Live validation command** — explicitly marked in the task with `network policy: limited-live`. This command may call the named public API once to verify current data structure compatibility. Its output is recorded in `docs/PROGRESS.md` as evidence but does not block automated test passes.
3. Live validation results are never baked into test expectations. If a live response differs from a fixture, the fixture or the adapter is investigated — tests are not rewritten to match transient API behaviour.

## No Customer Credentials

- FlexSavvy must never request, store, or transmit customer API credentials, OAuth tokens, supplier login details, or any authentication material tied to an individual user account.
- If a public API introduces authentication requirements in the future, this is a decision gate requiring human review and a new ADR.

## No Private Data in Fixtures

Committed fixture files must never contain:

- Real meter identifiers (MPANs, device IDs, serial numbers)
- Customer names, addresses, or postcodes
- Actual consumption data from identifiable households without explicit consent and anonymisation
- Supplier billing amounts or account numbers
- Smart-meter filenames that reveal customer information
- Any data that could be used to identify an individual or household

When in doubt, generate synthetic data instead of including real samples. Private validation datasets belong exclusively in `private-data/` and are excluded via `.gitignore`.
